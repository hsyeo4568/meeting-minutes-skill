# Onboarding 인터뷰 (에이전트 실행 스크립트)

> `config.yaml`이 없으면 스킬은 **이 인터뷰를 먼저 실행**한다. 폼을 미리 채우라고 시키지 말 것 —
> 아래 순서로 **한 번에 하나씩** 묻고, 답을 받은 뒤 다음으로. 끝나면 답으로 `config.yaml` +
> `profiles/<사용자>/{structure,contacts,domain-glossary,conventions}.md`를 생성한다.
>
> 원칙:
> - **형태를 정해주지 않는다.** Q4(어떤 회의)의 답에 따라 Q5~8(구조)을 묻는다. 회의 종류마다 반복.
> - 선택 그룹(5: 도메인 용어)은 "지금/나중" 물어보고 건너뛸 수 있게.
> - 모르거나 해당없으면 skip 허용 → 해당 산출물은 degradation(파일 fallback).
> - 답이 길면 그대로 인용해 프로필에 박고, 추측으로 채우지 말 것.

---

## 진행 방식
"회의록 스킬 셋업을 시작합니다. 몇 가지 물어볼게요 (대부분 선택, 모르면 건너뛰면 됩니다)." 안내 후 그룹 순서대로.

**배치 원칙 (한 문항씩 금지 — 온보딩 중단률 대책):** 그룹 1+2 필수 문항(Q1~Q4)은 **한 메시지에 묶어서** 질문. 회의별 상세(Q4a~Q4d)도 **회의당 한 메시지**로 묶음. 이후 그룹은 답변 기반으로 config/profile 초안을 **먼저 제안**하고(infer-then-confirm) 미확정 항목만 후속 질문 — 전 문항 순차 인터뷰 금지 (회의 3종이면 30~40턴 → 이탈).

## 그룹 1 — 정체성
- Q1. 회의록에서 본인을 뭐라고 표기할까요? (이름) → `config.identity.me`
- Q2. 소속 조직과 약어는? → `config.identity.org`

## 그룹 2 — 업무 맥락 (★ 구조의 뿌리)
- Q3. 주로 어떤 일을 하시나요? 한 줄로. → `config.project.name` 힌트 + 도메인 파악
- Q4. **정기적으로 하는 회의 종류를 알려주세요** (예: 데일리 이슈회의 / 주간 정기 / 1:1 / 디자인리뷰 / 고객미팅).
  - 사용자가 여러 회의를 한 번에 열거하면: "좋습니다 — {A}부터 하나씩 여쭐게요" 하고 회의별로 Q4a~4d 순회.
  - 각 회의마다 이어서:
    - Q4a. 이 회의 목적은? (한 줄)
    - Q4b. 참석 구성? (우리 조직 내부만 / 여러 조직 함께)
    - Q4c. 회의록을 어디로 공유? (팀챗 / 메일 / 위키·문서 / 안 함)
    - Q4d. **이 회의를 다른 회의와 어떻게 구분하나요?** (길이·주기·참석자·주제 등 신호 — 녹취만 보고 카테고리를 판별할 근거) → `structure.md` 판별 신호 섹션
  - → `config.categories` (회의별 산출물 플래그) + 각 회의 = 한 카테고리

## 그룹 3 — 회의별 구조 (★ Q4의 각 회의에 대해 반복)
- Q5. "{회의명}" 회의록에 어떤 섹션이 어떤 순서로 들어가면 좋을까요?
  - 막막해하면 녹취/설명 기반으로 **제안** → "이렇게 잡아봤습니다: A→B→C. 조정할까요?" (infer-then-confirm)
- Q6. Action Items를 어떻게 묶을까요? (조직별 / 담당자별 / 안 묶음 / 우선순위별)
- Q7. 안건 제목 스타일? (증상·정량 기반 / 안건명 그대로 / 주제 키워드)
- Q8. 이 회의에만 적용할 특수 규칙 있나요? (선택)
  - → `profiles/<사용자>/structure.md` (회의별 섹션·Action·제목·특수규칙)
- Q8b. (선택) 회의 종류와 무관한 **표기 관례**가 있나요? (호칭·직급 표기, 쓰면 안 되는 표현, 강조 규칙, 시간 표기 등)
  - → `profiles/<사용자>/conventions.md` — 없으면 빈 틀로 생성하고 "운영하며 교정 누적" 안내.

## 그룹 4 — 사람
- Q9. 회의에 참여하는 조직들과 약어는? → conventions.md / config 화자 라벨
- Q10. 자주 등장하는 핵심 인물(이름/역할)은? → contacts.md
- Q11. (메일 공유하는 회의가 있으면) 받는 사람/참조는 누구? → contacts.md 수신/참조 맵
- Q12. 데이터·자료를 요청할 외부 조직이 있나요? (다조직 협업·컨소시엄에서 로그/데이터를 타 조직에 요청하는 경우 — **해당 없으면 "없음"으로 건너뛰세요**) → contacts.md 외부 조직

## 그룹 5 — 도메인 용어 (선택 — "지금 채울까요, 나중에?")
- Q13. 자주 쓰는 약어 / Q14. STT가 자주 틀리는 인명 / Q15. 영역별 책임 조직 / Q16. 헷갈리는 용어쌍
  - → `profiles/<사용자>/domain-glossary.md`. 건너뛰면 빈 채로(교차검증 생략).

## 그룹 6 — 저장·도구·환경
- Q17. 완성 회의록을 어디에 보관하나요? (노트 vault / 폴더 / 위키 경로) → `config.paths.vault`
- Q18. 녹취·원본 파일이 있는 폴더는? → `config.paths.work_folder`
- Q19. 쓸 수 있는 도구는? (Slack / Gmail / 검색인덱서 / 지식그래프 — 없으면 다 파일로 떨굼) → `config.tools`, `config.channels`
- Q20. 작성 언어와 문체는? (예: 한국어 개조식 / 영어 / plain) → `config.locale`

---

## 마무리 (답 → 파일 생성)
1. `config.example.yaml`을 베이스로 `config.yaml` 작성 (답 매핑, 빈 값은 `null`/생략).
2. `profiles/_template/`를 `profiles/<사용자>/`로 복사 후 답으로 채움 — **structure.md가 핵심**(회의별 형태).
3. `config.project.profile`을 `profiles/<사용자>`로.
4. 검증: `python scripts/dry_run.py` → PASS 안내.
5. "셋업 완료. `/meeting-minutes [녹취파일]`로 시작하세요. 빠진 항목은 나중에 프로필 파일에서 보완 가능합니다."

> 무료(Claude Code 없는) 사용자: 이 인터뷰 대신 `PROMPT-ONLY.md`를 채팅에 붙여넣고, 프롬프트가 먼저 구조를 제안/확인한 뒤 작성.
